VAG Group ODIS Programming Requirements (2010–2026): A Cryptographic and Operational Analysis of Dealer Key Acquisition, Rolling Code Protocols, and Diagnostic Architecture
1. Executive Summary: The Transition from Hardware to Identity
The automotive security paradigm within the Volkswagen Auto Group (VAG)—encompassing Volkswagen, Audi, Bentley, Lamborghini, Seat, and Skoda—has undergone a fundamental metamorphosis over the sixteen-year period from 2010 to 2026. What was once a system defined by physical transponders and static PIN codes has evolved into a cloud-centric ecosystem governed by rigorous digital identity management and real-time server authentication. This report provides an exhaustive technical analysis of VAG’s Offboard Diagnostic Information System (ODIS) requirements for key programming and immobilizer adaptation, tracing the lineage from the hardware-based Component Security (CS) of the PQ platforms to the server-authenticated Vehicle Diagnostic Protection (SFD) of the MQB-Evo era.
The core thesis of this investigation posits that VAG has systematically dismantled the viability of "offline" programming through a dual strategy of cryptographic escalation and centralized authorization. In 2010, an immobilizer adaptation was a local handshake between control modules, facilitated by a static 5-digit PIN code often extractable via the OBD-II port. By 2026, the same operation has transformed into a complex cryptographic transaction authorized by the FAZIT database in Wolfsburg, requiring a technician to possess vetted credentials (GeKo/NASTF), a secure low-latency connection, and a pre-authorized "Dealer Key."
The concept of the "Dealer Key"—a transponder pre-coded with vehicle-specific serialization data at the factory—has become the central enforcement mechanism of this security model. While the aftermarket has developed sophisticated countermeasures involving microcontroller unit (MCU) memory extraction and server-side calculation, the window for non-dealer intervention is narrowing. The implementation of SFD (Vehicle Diagnostic Protection) and SFD2, driven by UNECE R155 cybersecurity regulations, introduces secure gateways that block unauthorized writing to control units, effectively serving as a digital firewall against independent repair.
This report serves as a definitive technical manual and strategic guide for security professionals, dealership technicians, and automotive engineers. It details the precise workflows within ODIS Service (ODIS-S) and ODIS Engineering (ODIS-E), the cryptographic underpinnings of Megamos AES and HITAG AES transponders, and the economic and operational implications of the current conflict between "Right to Repair" legislation and automotive cybersecurity imperatives.


  



2. The ODIS Ecosystem: Architecture, Connectivity, and Authentication
To understand key programming in the VAG ecosystem, one must first master the toolchain. The Offboard Diagnostic Information System (ODIS) is not merely software; it is a terminal interface for the Volkswagen Group's backend infrastructure. The distinction between the tool (ODIS) and the authorization system (GeKo) is the most critical concept for practitioners operating in this domain. Unlike generic OBD-II scanners that interact primarily with the vehicle's local bus, ODIS operates as a bridge between the vehicle's local network and VAG's central servers.
2.1 ODIS Service (ODIS-S) vs. ODIS Engineering (ODIS-E)
The dichotomy between ODIS Service and ODIS Engineering defines the operational capabilities of a workshop. While they share a common codebase for communication protocols (PDU-API), their user interfaces and permission sets are designed for fundamentally different user personas.
ODIS Service (ODIS-S) is the standard dealership interface designed for "Guided Fault Finding" (GFF). It is process-oriented, forcing the technician to follow a linear script.1 For key programming, ODIS-S is the only official pathway authorized by VAG for field technicians. It automates the connection to the backend, validates the GeKo credentials, and executes the Adapt Immobilizer test plan.3 It is designed to prevent user error but also restricts user freedom; a technician cannot simply "write a byte" to the cluster—they must execute a pre-approved server transaction. The software relies heavily on "Test Plans," which are scripted sequences of diagnostic commands that guide the user through complex procedures like key adaptation or module replacement.4
ODIS Engineering (ODIS-E), conversely, is intended for development, manufacturing, and deep-level engineering analysis. It provides direct access to the vehicle's control units via UDS (Unified Diagnostic Services) and KWP2000 services. It allows for raw memory reading/writing (ReadMemoryByAddress), direct EEPROM modification, and flashing (.sgo/.frf files) without the GFF wrapper.2 However, strictly speaking, ODIS-E cannot program keys in the modern era because key programming requires the calculation of encryption keys (MAC/CS) that are only generated by the FAZIT server during a secure session initiated by ODIS-S.6 The primary use of ODIS-E in the aftermarket is for reviving "bricked" modules or forcing firmware downgrades to vulnerable versions that allow aftermarket tools to extract security data. It allows engineers to bypass the logic checks of GFF, but it does not bypass the cryptographic requirements of the backend.
2.2 The Backend Trinity: FAZIT, GeKo, and SVM
The "online" requirement for VAG programming refers to three distinct backend systems that function in unison to secure the vehicle. Understanding these acronyms is essential for diagnosing authorization failures.
1. FAZIT (Fahrzeugauskunft- und zentrales Identifikations-Tool): This is the central identification tool and database located in Wolfsburg. FAZIT stores the "DNA" of every vehicle produced by the group. This includes the VIN, the Component Security (CS) bytes, the MAC synchronization codes, and the unique Transponder IDs (T-IDs) of all authorized keys.6 When ODIS requests to adapt a key, it is not asking the car to learn a new key; it is asking FAZIT to verify the key's pre-coded ID and release the sync data to the car's immobilizer. If the key's ID is not registered in FAZIT against that specific VIN, the server will deny the adaptation request.
2. GeKo (Geheimnis und Komponentenschutz): This is the user access management system. It translates to "Secret and Component Protection." To perform any security-relevant task (key learning, removing component protection, resetting immobilizers), the user must authenticate with a GeKo user ID and a SecurID token (rolling code 2FA).7 Without a valid GeKo login, ODIS-S functions are limited to reading DTCs and basic adaptations; the "Immobilizer Adaptation" menu will return specific error codes such as ODS9212E: Login could not be completed.6 GeKo is the gatekeeper that ensures only authorized personnel can access FAZIT.
3. SVM (Software Version Management): While primarily for coding and flashing, SVM ensures the vehicle configuration matches the factory build. During key programming, SVM checks often run in the background to ensure the BCM and Cluster software levels are compatible with the immobilizer strategy. If a technician has retrofitted a used cluster or BCM without properly updating the SVM record, the key programming process may abort due to a "Hardware/Software Mismatch."


  



2.3 Hardware Interfaces: VAS 6154 vs. J2534 PassThru
The physical link to the vehicle has evolved significantly, impacting the reliability of key programming operations. The standard dealer interface is the VAS 6154 (A or B variants), which natively supports DoIP (Diagnostics over IP). This protocol is mandatory for newer MLB-Evo and MQB-Evo platforms where high-speed data transfer is required to unlock the SFD gateway within the requisite time window.7
However, a significant development for the independent market is VAG's support for J2534 PassThru devices. ODIS-S can technically run with high-quality J2534 tools (e.g., DrewTech Cardaq, Autel MaxiFlash, Mongoose Plus).8 While this opens the door for independent shops to use OEM software without buying the $2,000+ VAS interface, there are critical limitations:
* Latency and Timing: ODIS generates high packet-per-second (PPS) bursts during topology scans and SFD unlocks. Inferior J2534 devices often introduce latency that causes the PDU-API (Protocol Data Unit API) to timeout. Since the FAZIT connection relies on strict session timers, a dropped packet can invalidate the entire authentication session, forcing the technician to restart the procedure and burn another GeKo token.7
* DoIP Requirement: For 2020+ vehicles (Golf 8, A3 8Y), a DoIP-capable VCI is mandatory. Standard CAN interfaces cannot communicate with the high-speed Ethernet bus used for the main gateway in these architectures. Attempts to program keys on an MQB-Evo platform using a standard CAN J2534 device will result in a failure to detect the gateway or an inability to execute the high-speed handshake required for SFD unlocking.

3. Platform-Specific Programming Analysis: A Chronological Deep Dive
The procedure and requirements for key programming vary drastically by platform. This section dissects the four major epochs of VAG security, analyzing the shift from local PIN codes to distributed server-side authentication.
3.1 PQ35/PQ46 Platform (2010–2015)
This era represents the transition from legacy systems to modern component security. It includes high-volume models like the VW Golf Mk6, Passat B7, Jetta Mk6, and the first-generation Tiguan.
* Immobilizer System: Immobilizer IV (Immo 4).
* Key Tech: Megamos Crypto 48 (Glass ID48).11
* Security Structure:
   * Component Security (CS): 7 Bytes.
   * MAC: None.
   * Dealer Key Requirement: Keys must be "pre-coded" with the 7-byte CS specific to that car before they can be adapted. A blank ID48 chip will be rejected by the cluster because it lacks the specific CS "personality" of the vehicle.12
* ODIS Procedure:
   1. Technician enters Special Functions > Immobilizer > Adapt Keys.
   2. ODIS requests GeKo login to retrieve the 5-digit PIN and the 7-byte CS from FAZIT.
   3. User inputs the number of keys.
   4. The Cluster (Address 17) enters learning mode, displaying "0-1", "1-2" on the odometer LCD.
   5. Technician cycles the ignition with each key.
* Aftermarket Viability: High. This generation is widely supported by aftermarket tools (VVDI2, Abrites, Autel). These tools function by reading the Immo data (CS + PIN) from the cluster via OBD or by reading the 24C32/24C64 EEPROM directly. Once the 7-byte CS is known, a "Dealer Key" can be generated on a bench programmer using a generic blank ID48 chip, effectively bypassing the need to order a pre-coded key from the dealer.13 The ease of this bypass is what drove VAG to develop the more robust Immo V system.
3.2 MQB Platform (2013–2019)
The introduction of the Modularer Querbaukasten (MQB) platform marked a seismic shift in VAG security.
* Models: VW Golf Mk7, Audi A3 8V, Skoda Octavia III, Seat Leon 5F.
* Immobilizer System: Immobilizer V (Immo 5) - Integrated primarily into the Dashboard (J285).
* Key Tech: Megamos AES (MQB48).
* Security Structure:
   * Component Security (CS): Extended to 16 Bytes and distributed across multiple modules.
   * Synchronization Data: 32 Bytes of alignment data.
   * Modules Involved: Cluster (Master), ECU, TCU (if DSG), ELV (Steering Lock), and Kessy (Keyless Access). All these modules must share the same CS and Sync data to authorize a start.13
* The "Dealer Key" Constraint: The most significant shift in MQB was the tightening of the Dealer Key definition. A key ordered from the dealer comes pre-programmed with the specific transponder ID authorized in the FAZIT database.
* ODIS Procedure:
   * Add Key / All Keys Lost (AKL): ODIS requires the "Order Key" process. The new key's ID is already provisioned in FAZIT. The adaptation simply "unlocks" the immobilizer to accept this pre-authorized ID. You cannot take a blank key and adapt it with ODIS unless that key was ordered by VIN from the Parts Department.14 The ODIS test plan validates the key's unique ID against the FAZIT allow-list before attempting adaptation.
* Aftermarket Viability:
   * Johnson Controls vs. VDO: Early MQB clusters made by VDO (Continental) allow for OBD reading of the NEC35xx processor to extract Immo Data. Clusters made by Johnson Controls (JCI) often require more invasive EEPROM reading or are unsupported via OBD.16
   * Virtual Cockpit: The introduction of the fully digital Virtual Cockpit (AID - Active Info Display) in later models (2016+) significantly hardened the system. These clusters typically block OBD reading methods entirely.
   * Sync Data Calculation: For "All Keys Lost" in the aftermarket, reading the CS from the ECU is insufficient. One requires the 32-byte Sync Data. This data is usually only available from the VAG server (which ODIS accesses). Third-party tools rely on "exploits" or server calculation services (like Xhorse or PolDiag) that charge per transaction to calculate this data from the cluster dump.17 This created a "pay-per-use" economy in the aftermarket key programming sector.
3.3 MLB-Evo Platform (2016–2026)
The MLB-Evo platform (Modular Longitudinal Matrix) serves high-end vehicles and introduces a centralized security architecture centered on the Body Control Module (BCM2) rather than the instrument cluster.
* Models: Audi A4 B9, A5 F5, Q7 4M, Q5 FY, Porsche Cayenne E3 (shared architecture).
* Immobilizer System: Immobilizer V - BCM2 Centric.
* Key Tech: HITAG AES / HITAG Pro.
* Security Structure:
   * Master Module: Unlike MQB, the BCM2 (Body Control Module 2) is the immobilizer master. The Dashboard is merely a display node.
   * Rolling Code: Highly aggressive rolling codes are implemented between the Key and BCM2, making "sniffing" or replay attacks extremely difficult.
* Key Programming Nuance:
   * Dealer Key: Strictly required for ODIS. The key must be ordered from the factory. The lead time for these keys can be days or weeks, as they are programmed at the facility in Germany or a regional center before shipping.
   * Aftermarket "Dealer Key": To create a key in the aftermarket, one must extract the BCM2 data. However, the BCM2 in MLB-Evo is encrypted.
   * 2021+ Lock: Audi updated the BCM2 firmware in late 2020/2021 to block the reading method used by aftermarket tools (like the "ABPROG" or "XP400"). For these newer models, even aftermarket tools often fail to read the encrypted BCM2, forcing the car to the dealer.18
* Specific Case - Audi Q7 (4M): The Q7 4M (2017+ North America) presents a distinct challenge. Early models (2017-2019) are susceptible to aftermarket BCM2 reading if the software version is low. Late models (2020+) are essentially dealer-only for "All Keys Lost" due to the locked BCM2.20
3.4 MQB-Evo & SFD (2020–2026)
The current generation, MQB-Evo, represents the pinnacle of VAG's "Connected Car" security strategy.
* Models: VW Golf Mk8, Audi A3 8Y, Seat Leon Mk4, Skoda Octavia Mk4, Cupra Formentor.
* Immobilizer System: Immobilizer V + SFD (Vehicle Diagnostic Protection).
* Key Tech: Megamos AES / HITAG AES (depending on Kessy).
* The SFD Barrier:
   * SFD (Schutz Fahrzeug Diagnose) is a layer above the immobilizer. It prevents any writing (coding/adaptation) to control units unless a secure token is provided.
   * Mechanism: When ODIS attempts to access the Adaptation channel to initiate key learning, the Gateway (J533) generates a challenge. ODIS sends this challenge to the VAG portal, which returns a signed Token. This Token is valid for 90 minutes.
   * Impact on Key Programming: You cannot even enter the key adaptation channel without an SFD unlock. This effectively kills "offline" diagnostic tools that cannot communicate with VAG's servers.
* Aftermarket Status:
   * Reading: The clusters use NEC35xx processors (e.g., R7F7014xx) that are securely locked. As of 2024/2025, tools like "PolDiag" and "Xhorse" claim support by physically soldering to the processor or using specialized adapters, but the success rate is variable and risks bricking the cluster.21
   * Writing: Even if data is read, writing back requires bypassing SFD. Aftermarket tools use "SFD Tokens" purchased from third-party servers or officially obtained via NASTF (in the US), but the integration is clunky compared to ODIS.23

4. Detailed Procedures: Dealer vs. Aftermarket Workflows
The divergence between the official ODIS procedure and the aftermarket method is stark. This comparison highlights the "Pre-Coded" vs. "Generated" key paradigm and the operational risks associated with each.
4.1 Dealer Workflow (ODIS Online)
This process assumes the workshop has a valid GeKo account and the key has been ordered by VIN from the Parts Distribution Center (PDC). It is a "clean" process with minimal hardware risk but high administrative overhead.
1. Preparation:
   * Battery maintainer connected (>12.5V) to prevent voltage drop during flash/adaptation.
   * All keys (old and new) present.
   * VAS 6154 connected via USB (preferred) or stable WiFi.
   * ODIS-S launched, "Diagnosis" mode selected.
2. Guided Fault Finding (GFF):
   * Scan vehicle. Confirm communication with Address 25 (Immobilizer) or 17 (Dashboard) or 46 (BCM2) depending on platform.
   * Navigate to: Special Functions > Immobilizer > Adapt Keys (or Components > Immobilizer > Adapt Key).3
3. Authorization:
   * ODIS prompts for GeKo login. Technician enters username/password.
   * Technician enters the rolling code from the SecurID RSA token.
   * ODIS connects to FAZIT. If the connection is successful, the FAZIT server returns a "ready" status.
4. Adaptation:
   * Prompt: "How many keys to adapt?" (Enter total number).
   * Prompt: "Insert Key 1 into reader coil/ignition." (For Kessy, hold against the marked spot on the steering column—often marked with a key symbol).26
   * ODIS writes the Key ID to the Immobilizer. FAZIT verifies the Key ID belongs to this VIN.
   * Wait for "Key Adapted" message on dash (e.g., 1/2).
   * Repeat for Key 2.
5. Finalization:
   * Immobilizer locks. System checks engine start permission.
   * DTCs cleared.
   * Transaction logged in FAZIT history.
4.2 Aftermarket Workflow (The "Generation" Method)
This process assumes the technician has a blank, unlocked key and is using a tool like Autel IM608 or VVDI2. It relies on forensic data extraction rather than administrative authorization.
1. Data Extraction (The Hard Part):
   * MQB: Attempt to read Immo Data via OBD. If failed (Locked Cluster), remove cluster, solder wires to the NEC processor points, and read the dump on a bench.27
   * MLB-Evo: Remove BCM2. Connect programmer to BCM2 pins (often requiring lifting pins or cutting traces on older methods, though newer adapters use friction probes). Read encrypted data. Upload to third-party server to calculate Immo Data.17
2. Dealer Key Generation:
   * Once CS (16 Bytes) and Sync Data (32 Bytes) are obtained, load a blank aftermarket key into the key programmer coil.
   * Select "Make Dealer Key." The tool writes the specific CS/Sync data onto the blank key's transponder. This effectively "clones" the identity of a key that would have been ordered from the factory.
3. Adaptation:
   * Reinstall Cluster/BCM2.
   * Use the tool to enter "Key Learning" mode.
   * Since the tool now knows the PIN/CS, it authorizes the login itself (bypassing GeKo in some cases) or uses the extracted data to satisfy the immobilizer's request.
   * Note: For MQB-Evo/SFD vehicles, the tool must also acquire an SFD token to perform this step, often requiring a separate "Unlock" fee or subscription.24

5. Economic and Regulatory Barriers: The GeKo/NASTF Wall
The technical ability to program a key is increasingly secondary to the administrative ability to access the car. The barriers to entry for key programming are no longer just the cost of the tool, but the cost of compliance and identity verification.
5.1 The Cost of Compliance (Dealer Perspective)
For a dealership, the cost is structural and integrated into their franchise agreement.
* ODIS Subscription: The "ODIS Service Flat Rate" (via erWin) can cost ~$1,500/year or ~$250/month per diagnostic laptop.28
* Key Cost: A genuine "Dealer Key" for an MQB/MLB vehicle typically retails between $300 and $800 depending on the model (e.g., Audi Q8 keys are significantly more expensive than Golf keys due to the advanced finishing and transponder tech).29
* Labor: Dealerships typically charge 1-2 hours of labor for the adaptation, primarily to cover the GFF scan time and GeKo administration.
5.2 The Independent Locksmith's Burden (NASTF VSP)
In the United States, accessing the GeKo system requires a Vehicle Security Professional (VSP) credential from the National Automotive Service Task Force (NASTF). This creates a regulated environment where only vetted individuals can perform security work.
* VSP Registry: The locksmith must pass a background check ($75) and pay a two-year subscription ($435).31
* Transaction Fees: While NASTF membership is free, specific "Immobilizer Code" requests or "Key Code" requests may incur per-transaction fees depending on the OEM portal (Audi/VW erWin sometimes charges per code).
* The "Audit" Risk: VAG is notoriously strict with GeKo usage. If a GeKo account is used to program a key for a vehicle that is later reported stolen, the logs in FAZIT (which record the GeKo User ID, IP address, and time) are used to trace the technician. This liability discourages many independents from performing "All Keys Lost" work on high-value models like the Bentley Continental GT or Audi Q7.33
5.3 The "Right to Repair" Tension
The implementation of SFD (and the upcoming SFD2) is often cited by VAG as a cybersecurity necessity under UNECE WP.29 R155. This regulation mandates that OEMs secure vehicle architectures against cyberattacks. However, the aftermarket views SFD as a mechanism to lock out independent repair.34
* SFD mechanism: Unlike Immo PINs, SFD does not require a "secret" in the traditional sense; it requires a permission (Token) granted by the OEM. This shifts the power dynamic: VAG can theoretically deny an SFD token to an aftermarket tool even if the technician is legitimate, citing "security concerns" or "unauthorized equipment."
* Workarounds: Tools like OBDeleven and some Autel scanners have implemented "Auto-Unlock" features by partnering with VAG or using authorized tokens, but these are often limited to coding and basic settings, not high-level immobilizer adaptation which remains strictly gated.23

6. Deep Dive: High-Value Case Studies
6.1 Bentley Continental GT (2018+ / MSB Platform)
The transition of the Bentley Continental GT to the MSB (Modularer Standardantriebsbaukasten) platform (shared with Porsche Panamera) introduced significant hurdles for independent service.
* Legacy (Pre-2018): Used a KESSY system similar to the VW Phaeton/Audi A8 D3. Address 05 (Access/Start). Aftermarket could often handle this via OBD using tools that exploited the older Kessy vulnerability.
* Modern (2018+): The immobilizer is tightly integrated into the BCM2 and the ECU. The encryption keys are 128-bit AES and the component protection is aggressive.
* Programming: Requires ODIS-S with the Bentley brand add-on. The keys are prohibitively expensive ($600+) and must be ordered by VIN.33
* Aftermarket: Extremely limited. Autel IM608 claims support for "All Keys Lost" on some models, but reports suggest high failure rates on the newest encrypted BCM versions.35 The risk of bricking a $2,000+ BCM2 makes dealer service the preferred route for most owners.
6.2 Porsche Cayenne (E3 / 2019+)
The Cayenne E3 shares the MLB-Evo architecture with the Audi Q7 4M but uses Porsche's specific PIWIS III diagnostic standard, which is distinct from ODIS in interface but similar in backend logic (PPN - Porsche Partner Network).
* Key Difference: While VAG uses ODIS, Porsche requires PIWIS. However, the underlying encryption (HITAG AES) is the same.
* Locksmith Status: "All Keys Lost" on a 2019+ Cayenne is virtually impossible for the aftermarket via OBD. The BCM must be removed, and even then, the encryption is robust. Most locksmiths will decline this job or refer to the dealer, citing the risk of damaging the BCM during extraction.36
6.3 VW Golf Mk8 (MQB-Evo)
The "Poster Child" for the new restrictions, the Golf Mk8 represents the current state-of-the-art in mass-market vehicle security.
* Cluster: VDO with NEC35xx (Locked).
* Immobilizer: 5th Gen with SFD.
* Situation: You cannot adapt a key without ODIS Online unless you physically hack the cluster to read the Immo Data.
* 2024 Breakthrough: Some tool manufacturers (e.g., Xhorse, ACDP) released "Solder-Free" adapters and "Interface Board" replacements for the Golf 8 cluster in late 2023/2024. This involves physically removing the processor or clamping onto it to bypass the lock bits, extracting the Immo Data, and then generating a dealer key. It is a high-skill, invasive procedure that voids warranties and carries significant risk.21

7. Future Outlook: 2026 and Beyond
As we approach 2026, the trajectory of VAG security is clear: Virtualization. VAG is moving towards "Functions on Demand" where the immobilizer is not just a theft device, but a digital rights management (DRM) tool.
* Cloud-Based ECUs: Future architectures (SSP 2.0 / MEB+) are expected to move the "Master" immobilizer logic entirely to the cloud, making "offline" key programming a conceptual impossibility. The car would verify the key via a cellular handshake with the server, not a local check.
* Implication: The "Aftermarket Dealer Key" market will likely collapse for these newer models, leaving only ODIS (or its successor) and authorized partners (NASTF VSP) as the viable service providers. This centralized control will likely face continued regulatory scrutiny under Right to Repair laws, but the technical reality will favor the OEM.
8. Conclusion
For the period of 2010–2026, VAG ODIS programming requirements have shifted from a "tool-centric" model to an "identity-centric" model.
* 2010-2015: The era of the Tool. If you had the right cable (VCDS, ODIS), you could do the job.
* 2015-2020: The era of the Data. You needed the tool, but you also needed the data (CS bytes, Pre-coded keys). The aftermarket survived by extracting this data.
* 2020-2026: The era of the Identity. You need the tool, the data, and the authorized identity (GeKo/SFD Token). The aftermarket can only survive by either hacking the hardware (invasive) or becoming an authorized participant (VSP).
The recommendation for professional workshops is dual-track: maintain a robust ODIS/GeKo setup for newer (2020+) and high-value vehicles to mitigate liability and ensure success, while utilizing advanced aftermarket tools (Autel/Xhorse) for the lucrative and accessible fleet of 2010–2019 vehicles where ODIS is often slower and more expensive.
